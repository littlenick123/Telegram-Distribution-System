# Telegram 私密影视库定时分发系统

使用 Telegram 私密频道保存影视媒体组，服务实时入库，并按各目标频道的每日时刻复制发布。目标帖子不显示“转发自”。系统还会在目标队列低于独立阈值时，通过管理机器人发送每日提醒。

## 环境要求

- Python 3.11 或更高版本
- 从 [my.telegram.org](https://my.telegram.org/) 获取的 `api_id` 与 `api_hash`
- 一个普通 Telegram 用户账号：必须加入所有源频道，并在所有目标频道拥有发帖权限
- 一个通过 BotFather 创建的管理机器人
- 所有管理员先私聊管理机器人发送 `/start`，机器人才能主动发送库存提醒

> Telegram 会话文件等同于登录凭据。请保护 `DATA_DIR`，不要上传 `.env`、数据库或 `.session` 文件。

## 安装

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
Copy-Item .env.example .env
```

Linux/macOS 使用 `python3 -m venv .venv` 和 `source .venv/bin/activate`。

编辑 `.env`：

```dotenv
TG_API_ID=123456
TG_API_HASH=你的_api_hash
TG_PHONE=+8613800000000
MANAGER_BOT_TOKEN=123456:你的机器人_token
ADMIN_USER_IDS=你的用户ID,另一位管理员ID
DATA_DIR=data
LOG_LEVEL=INFO
```

`ADMIN_USER_IDS` 是 Telegram 数字用户 ID，多个 ID 用英文逗号分隔。

## 首次启动

先登录普通用户账号：

```powershell
python -m app login
```

按照提示输入 Telegram 验证码和两步验证密码。然后启动服务：

```powershell
python -m app run
```

服务会在 `DATA_DIR` 中创建 SQLite 数据库和两个 Telethon 会话。首次添加源频道时只记录当前最新消息水位，不导入旧帖子；以后重启会补录该水位之后的媒体组。

## 推荐配置顺序

1. 管理员私聊机器人发送 `/start`。
2. `/source_add -1001234567890`
3. `/route_add -1001234567890 -1009876543210`
4. `/schedule_add -1009876543210 09:00 20:30`
6. `/threshold -1009876543210 24`
7. `/alert_time 09:00`
8. `/status` 和 `/alert_status` 检查结果。

频道 ID 通常以 `-100` 开头。用户账号必须已经能在 Telegram 中访问对应私密频道。

## 管理机器人命令手册

### 使用规则

- 命令只能在与管理机器人的**私聊**中使用；群组或频道中的命令会被忽略。
- 只有 `.env` 的 `ADMIN_USER_IDS` 白名单用户可以操作，其他用户只会收到“未授权”。
- 每位管理员必须先私聊机器人发送 `/start`，否则机器人可能无法主动发送低库存提醒。
- `<参数>` 表示必填参数，`[参数]` 表示可选参数；发送命令时不要输入尖括号或方括号。
- 频道 ID 建议使用 `-100` 开头的完整数字 ID，例如 `-1001234567890`。
- 时间必须使用 24 小时制 `HH:MM`，统一按 `Asia/Shanghai` 时区解释。
- 命令不区分大小写，也兼容 Telegram 自动附加的 `@机器人用户名`。

### 输入 `/` 显示命令菜单

服务每次启动后会自动把当前支持的全部管理命令同步到 Telegram 私聊命令菜单。进入机器人私聊并在输入框键入 `/`，即可看到命令下拉列表及中文用途说明，不需要在 BotFather 中手工执行 `/setcommands`。

- 菜单仅注册到机器人私聊作用域，群组或频道不使用这套菜单。
- 菜单对所有私聊用户可见，但命令执行仍受 `ADMIN_USER_IDS` 白名单保护；未授权用户无法操作服务。
- `/status`、`/source_list` 等无参数命令可从菜单直接选择并执行。
- `/source_add`、`/schedule_add` 等需要参数的命令从菜单选择后，会返回该命令的完整格式和示例；需要再发送包含实际频道 ID、时间或投递 ID 的完整命令。
- 代码中的命令定义是菜单的唯一来源，并会覆盖相同私聊作用域下通过 BotFather 配置的菜单。新增或修改命令后重启服务即可同步。
- Telegram 客户端可能缓存旧菜单；重启服务后如果没有立即更新，请重新进入机器人私聊或再次输入 `/`。
- 菜单同步失败不会中断影视库采集和发布，具体原因会写入服务日志。

### 帮助命令

#### `/start`

登记管理员与机器人的首次私聊，先发送欢迎说明和首次配置顺序，然后自动发送完整的分类命令帮助。它不会启动影视库服务；常驻服务仍需通过 `python -m app run` 启动。

```text
/start
```

#### `/help`

自动连续发送五段分类帮助，分别说明使用规则、源与路由、发布时间、库存提醒以及故障处理。每条命令都带用途、语法和示例。

```text
/help
```

### 源频道命令

源频道是保存原始影视媒体组的私密频道。系统只收录由至少两项媒体组成的 Telegram 媒体组，单个媒体和纯文本帖子会被忽略。

#### `/source_add <频道ID>`

添加或重新启用一个源频道。用户账号必须已经加入并能够读取该频道。

```text
/source_add -1001234567890
```

执行时系统会读取该频道当前最新消息 ID 作为采集水位，因此：

- 添加之前已经存在的帖子不会入库。
- 添加成功之后的新媒体组才会进入数据库。
- 如果频道已经存在，这条命令会更新频道名称并重新启用，但不会重置原采集水位。

#### `/source_list`

列出全部源频道，包括频道名称、频道 ID、启用状态和当前消息水位。

```text
/source_list
```

状态图标：`✅` 表示启用，`⏸` 表示停用。

#### `/source_enable <频道ID>`

重新启用一个已添加的源频道。重新启用后，周期补录会从原水位继续扫描，因此停用期间源频道新增的媒体组也可能被补录。

```text
/source_enable -1001234567890
```

#### `/source_disable <频道ID>`

暂停该源频道的采集和对应目标的计划发布。数据库中的频道、媒体组和待发队列不会删除。

```text
/source_disable -1001234567890
```

### 路由命令

路由定义“哪个源频道的内容发送到哪个目标频道”。一个源频道可以映射多个目标，但同一个目标频道不能同时映射到多个源频道。

#### `/route_add <源频道ID> <目标频道ID>`

建立或重新启用一条源到目标的映射。

```text
/route_add -1001234567890 -1009876543210
```

要求：

- 源频道必须已经通过 `/source_add` 添加。
- 用户账号必须能够访问目标频道。
- 对于广播频道，用户账号必须是创建者或拥有“发布消息”权限的管理员。
- 路由建立后，仅新入库的媒体组会进入这个目标的队列；以前已经入库的媒体组不会自动补入。

#### `/route_list`

列出全部源到目标映射及其启用状态。

```text
/route_list
```

#### `/route_enable <目标频道ID>`

重新启用目标映射。此前已经进入该目标的 `pending` 队列会继续发布；路由停用期间新入库的媒体组不会补入该目标。

```text
/route_enable -1009876543210
```

#### `/route_disable <目标频道ID>`

暂停该目标的计划发布，并停止为新媒体组创建该目标的投递记录。已有队列和发布历史不会删除，低库存提醒也会暂停。

```text
/route_disable -1009876543210
```

#### `/route_backfill <目标频道ID> all`

将目标路由对应源频道在数据库中的全部有效媒体组补入该目标的 `pending` 队列。

```text
/route_backfill -1009876543210 all
```

- 包括已经在其他目标频道发布过的媒体组。
- 不扫描首次添加源频道之前的 Telegram 历史，只使用当前数据库已有内容。
- 已经存在该目标投递记录的媒体组保持原状态，不会重置 `sent`、`failed`、`cancelled` 或 `ambiguous`。
- 重复执行不会重复入队，并会返回新增、已存在和当前 `pending` 数量。
- 禁用路由也可以回填，但只有重新启用后才会按计划发布。

### 发布时间命令

每个目标可以配置多个每日发布时间。每个计划时刻最多发布一组媒体；队列为空时直接跳过。服务停机错过的有效时刻会在恢复后依次补齐，全局串行发送，同一目标两次成功发布至少间隔 60 秒。

#### `/schedule_add <目标频道ID> <HH:MM> [HH:MM ...]`

为目标频道一次添加一个或多个每日发布时间，并返回新增及已存在的计划 ID。

```text
/schedule_add -1009876543210 09:00
/schedule_add -1009876543210 09:00 13:00 20:30
```

目标频道必须已经建立路由。同一批中的重复时间会自动去重；同一目标、同一时刻重复添加不会创建重复计划。系统会先验证整批时间，任一时间格式无效都不会写入。当天已经过去的时刻不会因为刚添加计划而立即补发。

#### `/schedule_list [目标频道ID]`

不带参数时列出全部计划；提供目标频道 ID 时只显示该目标。结果包含计划 ID、目标频道、状态和每日时刻，并按目标和时间排序。

```text
/schedule_list
/schedule_list -1009876543210
```

示例输出中的 `#3` 表示计划 ID 为 `3`。

#### `/schedule_del <计划ID>`

永久删除一个发布时间，但不会删除目标路由、待发队列或已经发布的记录。

```text
/schedule_del 3
```

#### `/schedule_del <目标频道ID> <HH:MM> [HH:MM ...]`

一次删除指定目标的一个或多个发布时间，不影响该目标的其他时间。

```text
/schedule_del -1009876543210 09:00 13:00
```

系统会返回实际删除和未找到的时间。任一时间格式无效时整批不执行。

#### `/schedule_del <目标频道ID> all`

删除指定目标频道的全部发布时间，但不删除路由、待发库存或发布历史。

```text
/schedule_del -1009876543210 all
```

为避免误删，仅发送 `/schedule_del -1009876543210` 不会执行，必须明确提供时间或 `all`。

### 库存与提醒命令

库存只统计状态为 `pending` 且源媒体组仍完整的投递。`sending`、`sent`、`failed`、`cancelled` 和 `ambiguous` 均不计入可用库存。

#### `/threshold <目标频道ID> <数量>`

修改单个目标频道的低库存阈值。数量必须是大于 `0` 的整数；默认阈值为 `24`。

```text
/threshold -1009876543210 24
```

每日检查时，仅当 `pending 数量 < 阈值` 才会提醒；如果刚好等于阈值则不提醒。

#### `/alert_enable <目标频道ID>`

开启指定目标频道的每日低库存提醒。

```text
/alert_enable -1009876543210
```

#### `/alert_disable <目标频道ID>`

关闭指定目标频道的低库存提醒，不影响采集、排队或正常发布。

```text
/alert_disable -1009876543210
```

#### `/alert_time <HH:MM>`

修改所有目标共用的每日库存检查时间，默认是 `09:00`。

```text
/alert_time 09:00
```

低库存目标会合并成一条消息发送给所有管理员。系统同一自然日只检查并汇总一次；如果当天检查已经完成，修改时间不会在当天触发第二次提醒。

#### `/alert_status`

查看所有目标的当前 `pending` 数量、独立阈值以及告警开关状态。

```text
/alert_status
```

#### `/queue [目标频道ID]`

查看可发布库存。省略频道 ID 时显示全部目标；提供频道 ID 时只显示该目标。

```text
/queue
/queue -1009876543210
```

输出中的 `pending=18` 表示目标当前有 18 组可按计划发布的媒体组。

### 服务状态与故障命令

#### `/status`

显示服务的数据库状态摘要，包括：

- 当前启用的源频道数量。
- 当前启用的目标映射数量。
- 各投递状态的总数量。
- 每日库存检查时间。

```text
/status
```

它反映数据库和调度配置，不代表 Telegram 网络一定在线；网络故障应同时查看服务终端日志。

#### `/issues [目标频道ID]`

列出最近最多 50 条需要人工处理的 `failed` 或 `ambiguous` 投递。省略频道 ID 时查看所有目标。

```text
/issues
/issues -1009876543210
```

每条记录会显示投递 ID、状态、目标频道、源媒体组 `grouped_id` 和错误原因。后续 `/retry` 或 `/skip` 使用的是开头的 `#投递ID`。

#### `/retry <投递ID>`

把 `failed` 或 `ambiguous` 投递重新设置为 `pending`，等待后续计划时刻再次发送。

```text
/retry 42
```

对于 `ambiguous`，Telegram 可能已经收到上一次发送。重试前必须先人工检查目标频道，否则可能产生重复帖子。`sent`、`pending`、`sending` 或 `cancelled` 状态不能使用此命令重试。

#### `/skip <投递ID>`

将 `pending`、`failed` 或 `ambiguous` 投递标记为 `cancelled`，以后不再自动发送。

```text
/skip 42
```

该操作只影响指定目标的这条投递，不会删除源频道媒体组，也不会撤回其他目标已经发布的副本。`sending` 和 `sent` 状态不能跳过。

## 状态和故障处理

- `pending`：可发布库存，低库存统计只计算此状态。
- `sending`：正在发送。
- `sent`：发送成功。
- `failed`：确定发送失败，可用 `/retry` 重新入队。
- `cancelled`：源媒体组删除、不完整或被管理员跳过。
- `ambiguous`：发送过程中服务中断，Telegram 是否已接收无法确定。请先检查目标频道，再决定 `/retry` 或 `/skip`，避免重复。

## 测试

自动化测试不连接 Telegram：

```powershell
python -m unittest discover -v
```

正式使用前建议建立一个测试源频道和两个测试目标频道，验证媒体、逐项文案、权限、发布时间和告警私聊均符合预期。
