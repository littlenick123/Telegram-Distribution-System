# Telegram 私密影视库定时分发系统

使用 Telegram 私密频道保存影视媒体组，服务实时入库，并按各目标频道的每日时刻复制发布。目标帖子不显示“转发自”。系统还会在目标队列低于独立阈值时，通过管理机器人发送每日提醒。

## 环境要求

- Python 3.11 或更高版本
- 从 [my.telegram.org](https://my.telegram.org/) 获取的 `api_id` 与 `api_hash`
- 一个普通 Telegram 用户账号：必须加入所有源频道，并在所有目标频道拥有发帖权限
- 一个通过 BotFather 创建的管理机器人
- 所有管理员先私聊管理机器人发送 `/start`，机器人才能主动发送库存提醒

> Telegram 会话文件等同于登录凭据。请保护 `DATA_DIR`，不要上传 `.env`、数据库或 `.session` 文件。

## 安装

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
Copy-Item .env.example .env
```

Linux/macOS 使用 `python3 -m venv .venv` 和 `source .venv/bin/activate`。

编辑 `.env`：

```dotenv
TG_API_ID=123456
TG_API_HASH=你的_api_hash
TG_PHONE=+8613800000000
MANAGER_BOT_TOKEN=123456:你的机器人_token
ADMIN_USER_IDS=你的用户ID,另一位管理员ID
DATA_DIR=data
LOG_LEVEL=INFO
```

`ADMIN_USER_IDS` 是 Telegram 数字用户 ID，多个 ID 用英文逗号分隔。

## 首次启动

先登录普通用户账号：

```powershell
python -m app login
```

按照提示输入 Telegram 验证码和两步验证密码。然后启动服务：

```powershell
python -m app run
```

服务会在 `DATA_DIR` 中创建 SQLite 数据库和两个 Telethon 会话。源频道可选择只监控新帖、扫描全部历史，或从指定消息 ID 开始建立影视库；重启后会从持久化游标继续。

从旧版本升级时，服务启动会自动把数据库迁移到 schema v3。现有普通频道/话题路由、投递、发布时间和告警设置会保留；已有路由默认继续从原水位之后接收，不会突然获得旧库存。

## 推荐配置顺序

1. 管理员私聊机器人发送 `/start`。
2. `/source_add -1001234567890 all`（或指定消息 ID）
3. `/route_add -1001234567890 -1009876543210`（仅新内容），或在末尾追加 `all`/消息 ID 立即回填
4. `/schedule_add -1009876543210 09:00 20:30`
5. `/threshold -1009876543210 24`
6. `/alert_time 09:00`
7. `/status` 和 `/alert_status` 检查结果。

频道 ID 通常以 `-100` 开头。用户账号必须已经能在 Telegram 中访问对应私密频道。

## 管理机器人命令手册

### 使用规则

- 命令只能在与管理机器人的**私聊**中使用；群组或频道中的命令会被忽略。
- 只有 `.env` 的 `ADMIN_USER_IDS` 白名单用户可以操作，其他用户只会收到“未授权”。
- 每位管理员必须先私聊机器人发送 `/start`，否则机器人可能无法主动发送低库存提醒。
- `<参数>` 表示必填参数，`[参数]` 表示可选参数；发送命令时不要输入尖括号或方括号。
- 频道 ID 建议使用 `-100` 开头的完整数字 ID，例如 `-1001234567890`。
- 普通目标标识就是频道 ID；话题目标标识为 `群组ID:话题ID`，例如 `-1009876543210:135`。
- 时间必须使用 24 小时制 `HH:MM`，统一按 `Asia/Shanghai` 时区解释。
- 命令不区分大小写，也兼容 Telegram 自动附加的 `@机器人用户名`。

### 输入 `/` 显示命令菜单

服务每次启动后会自动把当前支持的全部管理命令同步到 Telegram 私聊命令菜单。进入机器人私聊并在输入框键入 `/`，即可看到命令下拉列表及中文用途说明，不需要在 BotFather 中手工执行 `/setcommands`。

- 菜单仅注册到机器人私聊作用域，群组或频道不使用这套菜单。
- 菜单对所有私聊用户可见，但命令执行仍受 `ADMIN_USER_IDS` 白名单保护；未授权用户无法操作服务。
- `/status`、`/source_list` 等无参数命令可从菜单直接选择并执行。
- `/source_add`、`/schedule_add` 等需要参数的命令从菜单选择后，会返回该命令的完整格式和示例；需要再发送包含实际频道 ID、时间或投递 ID 的完整命令。
- 代码中的命令定义是菜单的唯一来源，并会覆盖相同私聊作用域下通过 BotFather 配置的菜单。新增或修改命令后重启服务即可同步。
- Telegram 客户端可能缓存旧菜单；重启服务后如果没有立即更新，请重新进入机器人私聊或再次输入 `/`。
- 菜单同步失败不会中断影视库采集和发布，具体原因会写入服务日志。

### 帮助命令

#### `/start`

登记管理员与机器人的首次私聊，先发送欢迎说明和首次配置顺序，然后自动发送完整的分类命令帮助。它不会启动影视库服务；常驻服务仍需通过 `python -m app run` 启动。

```text
/start
```

#### `/help`

自动连续发送五段分类帮助，分别说明使用规则、源与路由、发布时间、库存提醒以及故障处理。每条命令都带用途、语法和示例。

```text
/help
```

### 源频道命令

源频道是保存原始影视媒体组的私密频道。系统只收录由至少两项媒体组成的 Telegram 媒体组，单个媒体和纯文本帖子会被忽略。

#### `/source_add <频道ID> [all|消息ID]`

添加或重新启用源频道。用户账号必须已经加入并能够读取该频道。

```text
/source_add -1001234567890
/source_add -1001234567890 all
/source_add -1001234567890 4348
```

三种模式：

- 不带范围：保持原行为，以当前最新消息为水位，只监控之后的新媒体组。
- `all`：后台从当前最新消息向最早可访问内容扫描全部历史。
- 数字消息 ID：包含该消息并扫描到当前最新；若起点位于媒体组中间会补齐整组。
- 历史起点只接受数字消息 ID，不接受完整 `https://t.me/...` 链接。
- 已有源再次指定更早范围时会扩展历史覆盖；唯一约束保证重复扫描不会重复媒体组。
- 单媒体和纯文本仍会忽略。命令立即返回，完成或失败后机器人私聊发起管理员。

#### `/source_list`

列出全部源频道，包括频道名称、频道 ID、启用状态、正向水位、历史最早覆盖位置、扫描状态和最近错误。

```text
/source_list
```

状态图标：`✅` 表示启用，`⏸` 表示停用。

#### `/source_enable <频道ID>`

重新启用一个已添加的源频道。重新启用后，周期补录会从原水位继续扫描，因此停用期间源频道新增的媒体组也可能被补录。

```text
/source_enable -1001234567890
```

#### `/source_disable <频道ID>`

暂停该源频道的采集和对应目标的计划发布。数据库中的频道、媒体组和待发队列不会删除。

```text
/source_disable -1001234567890
```

### 路由命令

路由定义“哪个源频道的内容发送到哪个目标频道或话题”。一个源频道可以映射多个目标；目标身份由“频道/群组 ID + 可选话题 ID”共同确定，因此同群多个话题可以拥有独立路由、队列、计划和库存阈值，但同一个具体话题不能同时映射到多个源频道。

#### `/route_add <源频道ID> <目标标识> [all|消息ID]`

建立或重新启用一条源到目标的映射，并可在创建时直接指定该目标的独立历史起点。

```text
# 普通目标：只接收建立路由后的新媒体组
/route_add -1001234567890 -1009876543210

# 普通目标：建立路由并回填全部历史
/route_add -1001234567890 -1009876543210 all

# 普通目标：建立路由并从源消息 4348（包含）开始
/route_add -1001234567890 -1009876543210 4348

# 话题目标：推荐使用组合目标标识
/route_add -1001234567890 -1009876543210:135
/route_add -1001234567890 -1009876543210:135 all
/route_add -1001234567890 -1009876543210:135 4348

# 兼容原有话题写法；第四个参数可选
/route_add -1001234567890 -1009876543210 135
/route_add -1001234567890 -1009876543210 135 all
/route_add -1001234567890 -1009876543210 135 4348
```

要求：

- 源频道必须已经通过 `/source_add` 添加。
- 用户账号必须能够访问目标频道。
- 对于广播频道，用户账号必须是创建者或拥有“发布消息”权限的管理员。
- 话题目标必须是已开启论坛话题功能的超级群组，且话题必须存在、未关闭、未隐藏。
- 话题 ID 是 Telegram 的数字话题 ID；本命令不接受消息链接。
- 不提供历史范围时，路由会记录源频道当时的最新消息 ID，默认仅该位置之后的新媒体组进入目标；正在进行的历史扫描不会污染这个新时间线。
- `all` 表示从最早可访问消息开始；数字消息 ID 表示从该源消息开始并包含边界所在的完整媒体组。
- 创建时指定历史范围，会立即把数据库中已有的有效媒体组入队；尚未覆盖的范围会在后台扫描，完成前只暂停这一个目标。
- 话题目标追加历史范围时，推荐使用 `群组ID:话题ID`，这样最后一个数字始终明确表示源消息 ID。
- 原有 `/route_add <源ID> <群组ID> <话题ID>` 保持兼容；还可追加第四个 `all` 或消息 ID。
- 已有路由以后需要继续向更早范围扩展时，使用 `/route_backfill <目标标识> all` 或数字消息 ID。

#### `/route_list`

列出全部源到目标映射、独立消息起点、回填状态及最近错误。

```text
/route_list
```

#### `/route_enable <目标标识>`

重新启用目标映射。此前已经进入该目标的 `pending` 队列会继续发布；路由停用期间新入库的媒体组不会补入该目标。

```text
/route_enable -1009876543210
/route_enable -1009876543210:135
```

#### `/route_disable <目标标识>`

暂停该目标的计划发布，并停止为新媒体组创建该目标的投递记录。已有队列和发布历史不会删除，低库存提醒也会暂停。

```text
/route_disable -1009876543210
/route_disable -1009876543210:135
```

#### `/route_del <目标标识>`

直接永久删除一个普通频道或话题目标映射，不需要额外的确认参数。

```text
/route_del -1009876543210
/route_del -1009876543210:135
```

删除内容包括该目标的全部投递记录（含已发布记录）、待发队列、发布时间、调度状态、库存阈值、告警设置和历史起点。源频道及其媒体组、同源其他目标，以及 Telegram 中已经发布的帖子不会删除。

命令会先自动停用目标。如果恰好有投递处于 `sending`，本次不会删除，目标会保持停用；等待发送结束后重新执行同一条 `/route_del` 即可。删除成功后可以使用 `/route_add` 为相同目标建立一条全新的时间线，并重新设置发布时间和阈值。

例如，把错误使用 `all` 的话题目标重新设置为从源消息 `14209` 开始：

```text
/route_del -1003176453321:2
/route_add -1001718011697 -1003176453321:2 14209
/schedule_add -1003176453321:2 14:46
/threshold -1003176453321:2 24
```

⚠️ 删除不可恢复，且不会撤回 Telegram 中已发出的帖子。重新添加时应使用明确的消息 ID；使用 `all` 仍表示从源频道最早可访问历史开始。

#### `/route_backfill <目标标识> <all|消息ID>`

为该目标设置独立历史起点，并把符合范围的有效媒体组补入其 `pending` 队列。

```text
/route_backfill -1009876543210 all
/route_backfill -1009876543210:135 all
/route_backfill -1009876543210 4348
/route_backfill -1009876543210:135 4348
```

- `all` 从最早可访问消息开始；数字 ID 表示包含该源消息及之后的内容。
- 普通目标使用频道 ID；话题目标使用 `群组ID:话题ID`。
- 每条路由的起点、FIFO 队列、计划和库存完全独立。
- 如果源影视库尚未覆盖该范围，命令会自动扩展源历史扫描范围。
- 较新起点的目标可先完成并恢复发布，较早目标继续后台扫描。
- 两个目标追完各自库存后都会继续接收未来新媒体组。
- 包括已经在其他目标频道发布过的媒体组。
- 已经存在该目标投递记录的媒体组保持原状态，不会重置 `sent`、`failed`、`cancelled` 或 `ambiguous`。
- 路由起点只能向更早移动；不能通过回填命令向后移动并删除已有库存。
- 重复执行相同起点不会重复入队，并会返回新增、已存在、当前库存和回填状态。
- 禁用路由也可以回填，但只有重新启用后才会按计划发布。

### 发布时间命令

每个目标可以配置多个每日发布时间。每个计划时刻最多发布一组媒体；队列为空时直接跳过。服务停机错过的有效时刻会在恢复后依次补齐，全局串行发送，同一目标两次成功发布至少间隔 60 秒。

#### `/schedule_add <目标标识> <HH:MM> [HH:MM ...]`

为目标频道一次添加一个或多个每日发布时间，并返回新增及已存在的计划 ID。

```text
/schedule_add -1009876543210 09:00
/schedule_add -1009876543210 09:00 13:00 20:30
/schedule_add -1009876543210:135 09:00 13:00 20:30
```

目标频道必须已经建立路由。同一批中的重复时间会自动去重；同一目标、同一时刻重复添加不会创建重复计划。系统会先验证整批时间，任一时间格式无效都不会写入。当天已经过去的时刻不会因为刚添加计划而立即补发。

#### `/schedule_list [目标标识]`

不带参数时列出全部计划；提供目标频道 ID 时只显示该目标。结果包含计划 ID、目标频道、状态和每日时刻，并按目标和时间排序。

```text
/schedule_list
/schedule_list -1009876543210
/schedule_list -1009876543210:135
```

示例输出中的 `#3` 表示计划 ID 为 `3`。

#### `/schedule_del <计划ID>`

永久删除一个发布时间，但不会删除目标路由、待发队列或已经发布的记录。

```text
/schedule_del 3
```

#### `/schedule_del <目标标识> <HH:MM> [HH:MM ...]`

一次删除指定目标的一个或多个发布时间，不影响该目标的其他时间。

```text
/schedule_del -1009876543210 09:00 13:00
/schedule_del -1009876543210:135 09:00 13:00
```

系统会返回实际删除和未找到的时间。任一时间格式无效时整批不执行。

#### `/schedule_del <目标标识> all`

删除指定目标频道的全部发布时间，但不删除路由、待发库存或发布历史。

```text
/schedule_del -1009876543210 all
/schedule_del -1009876543210:135 all
```

为避免误删，仅发送 `/schedule_del -1009876543210` 不会执行，必须明确提供时间或 `all`。

### 库存与提醒命令

库存只统计状态为 `pending` 且源媒体组仍完整的投递。`sending`、`sent`、`failed`、`cancelled` 和 `ambiguous` 均不计入可用库存。

#### `/threshold <目标标识> <数量>`

修改单个目标频道的低库存阈值。数量必须是大于 `0` 的整数；默认阈值为 `24`。

```text
/threshold -1009876543210 24
/threshold -1009876543210:135 24
```

每日检查时，仅当 `pending 数量 < 阈值` 才会提醒；如果刚好等于阈值则不提醒。

#### `/alert_enable <目标标识>`

开启指定目标频道的每日低库存提醒。

```text
/alert_enable -1009876543210
/alert_enable -1009876543210:135
```

#### `/alert_disable <目标标识>`

关闭指定目标频道的低库存提醒，不影响采集、排队或正常发布。

```text
/alert_disable -1009876543210
/alert_disable -1009876543210:135
```

#### `/alert_time <HH:MM>`

修改所有目标共用的每日库存检查时间，默认是 `09:00`。

```text
/alert_time 09:00
```

低库存目标会合并成一条消息发送给所有管理员。系统同一自然日只检查并汇总一次；如果当天检查已经完成，修改时间不会在当天触发第二次提醒。

#### `/alert_status`

查看所有目标的当前 `pending` 数量、独立阈值以及告警开关状态。

```text
/alert_status
```

#### `/queue [目标标识]`

查看可发布库存。省略频道 ID 时显示全部目标；提供频道 ID 时只显示该目标。

```text
/queue
/queue -1009876543210
/queue -1009876543210:135
```

输出中的 `pending=18` 表示目标当前有 18 组可按计划发布的媒体组。

### 服务状态与故障命令

#### `/status`

显示服务的数据库状态摘要，包括：

- 当前启用的源频道数量。
- 当前启用的目标映射数量。
- 各投递状态的总数量。
- 每日库存检查时间。

```text
/status
```

它反映数据库和调度配置，不代表 Telegram 网络一定在线；网络故障应同时查看服务终端日志。

#### `/issues [目标标识]`

列出最近最多 50 条需要人工处理的 `failed` 或 `ambiguous` 投递。省略频道 ID 时查看所有目标。

```text
/issues
/issues -1009876543210
/issues -1009876543210:135
```

每条记录会显示投递 ID、状态、目标频道、源媒体组 `grouped_id` 和错误原因。后续 `/retry` 或 `/skip` 使用的是开头的 `#投递ID`。

话题目标发布时，整组媒体会回复到配置的话题根消息，因此不会显示“转发自”。如果话题后来被关闭、删除、隐藏或账号失去发帖权限，该投递会进入 `failed` 并显示 Telegram 错误；系统绝不会把内容降级发送到群组默认区域。修复话题或权限后可使用 `/retry <投递ID>` 重新入队。

#### `/retry <投递ID>`

把 `failed` 或 `ambiguous` 投递重新设置为 `pending`，等待后续计划时刻再次发送。

```text
/retry 42
```

对于 `ambiguous`，Telegram 可能已经收到上一次发送。重试前必须先人工检查目标频道，否则可能产生重复帖子。`sent`、`pending`、`sending` 或 `cancelled` 状态不能使用此命令重试。

#### `/skip <投递ID>`

将 `pending`、`failed` 或 `ambiguous` 投递标记为 `cancelled`，以后不再自动发送。

```text
/skip 42
```

该操作只影响指定目标的这条投递，不会删除源频道媒体组，也不会撤回其他目标已经发布的副本。`sending` 和 `sent` 状态不能跳过。

## 状态和故障处理

- `pending`：可发布库存，低库存统计只计算此状态。
- `sending`：正在发送。
- `sent`：发送成功。
- `failed`：确定发送失败，可用 `/retry` 重新入队。
- `cancelled`：源媒体组删除、不完整或被管理员跳过。
- `ambiguous`：发送过程中服务中断，Telegram 是否已接收无法确定。请先检查目标频道，再决定 `/retry` 或 `/skip`，避免重复。

## 测试

自动化测试不连接 Telegram：

```powershell
python -m unittest discover -v
```

正式使用前建议建立一个测试源频道和两个测试目标频道，验证媒体、逐项文案、权限、发布时间和告警私聊均符合预期。
